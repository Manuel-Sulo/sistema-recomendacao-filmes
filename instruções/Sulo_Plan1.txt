# Sistema de Recomendação de Filmes — Plano Técnico Final

A base do projeto continua a mesma do Lab #04: **Angular no frontend**, **PHP puro no backend**, **SQL relacional**, troca de dados por **HTTP/JSON**, **autenticação completa**, **CRUD**, **API externa**, **modo claro/escuro**, **2 idiomas**, **exportação CSV/PDF**, e código organizado com separação real de responsabilidades. O PDF também deixa claro que existe liberdade nas funcionalidades do sistema, desde que ele seja útil, coerente e com fluxo lógico consistente. 

## 1) Conceito funcional do sistema

Para o tema **Sistema de Recomendação de Filmes**, a proposta mais sólida é um sistema onde o utilizador pesquisa filmes, guarda favoritos, avalia filmes, cria uma lista pessoal e recebe recomendações baseadas no seu histórico, gostos e géneros preferidos. Isto encaixa bem no objetivo do lab porque cria interação real, fluxo lógico e um problema claro a resolver dentro do tema. 

A proposta funcional não é imposta pelo PDF; é uma escolha técnica coerente com a liberdade que ele concede. 

---

## 2) O que o Lab obriga neste sistema

O sistema tem de cumprir obrigatoriamente:

* autenticação completa: registo, login, logout e recuperação de senha;
* base de dados relacional com pelo menos 3 tabelas e CRUD;
* integração com pelo menos uma API externa, quando aplicável;
* interface responsiva;
* modo claro e escuro;
* suporte a pelo menos 2 idiomas;
* exportação de dados ou relatórios em CSV ou PDF. 

Quando fizer sentido para o tema, também deve haver distinção entre utilizadores e permissões diferentes. Para filmes, isso pode ser justificado com `user` e `admin`.

## 4) Arquitetura técnica

### Frontend Angular

* `core/`: auth, guards, interceptors, config global;
* `shared/`: botões, cards, modais, tabelas, pipes, diretivas;
* `features/`: módulos por funcionalidade;
* `layouts/`: shell visual do app;
* `services/`: chamadas HTTP;
* `models/`: contratos de dados.

### Backend PHP puro

* `Controllers/`: recebem requests e devolvem JSON;
* `Services/`: regras de negócio;
* `Repositories/`: queries SQL;
* `Models/`: entidades;
* `Middleware/`: autenticação, autorização, CORS;
* `Validators/`: validação de input;
* `Helpers/`: respostas, utilitários, formatação.

Fluxo:
`Angular → HTTP JSON → PHP Controller → Service → Repository → SQL → Response JSON`

## 7) API externa: melhor escolha

A API mais forte para este sistema é a **TMDb API**. A documentação oficial indica que ela cobre **movie, tv, actor e image API**, permite **search/movie**, **movie details**, **discover/movie**, **genre list**, **recommendations**, **reviews** e **watch providers**. A própria FAQ diz que a API pode ser usada para aplicações com dados/imagens de filmes e que é gratuita para uso não comercial, com atribuição obrigatória. ([The Movie Database (TMDB)][1])

### Decisão técnica

**TMDb deve ser a API principal**, porque cobre exatamente o tipo de sistema que estás a montar: pesquisa, detalhe, géneros, recomendações e até dados de disponibilidade em streaming. ([The Movie Database (TMDB)][2])

### API secundária possível

A **OMDb API** também é útil como complemento, porque é um serviço REST para informação sobre filmes; a página oficial indica também que o **Poster API** existe, mas é restrito a patrons. Isso torna a OMDb mais indicada como apoio do que como fonte principal. ([omdbapi.com][3])

### Opcional avançado

A **JustWatch Partner API** suporta lookup por **TMDb ID** e **JustWatch ID**, e o site posiciona-se como base de dados de streaming. Porém, por ser partner API, o acesso não deve ser assumido como livre; para este projeto eu trataria isso como **[bloqueado]** até haver acesso confirmado. ([JustWatch][4])

## 8) Scraping: permitido ou não?

**[bloqueado]** O PDF não autoriza nem proíbe scraping explicitamente. Ele só exige integração com API externa quando aplicável. Portanto, não dá para afirmar que scraping está permitido como requisito do lab. 

### Recomendação técnica

Não usar scraping como base do projeto. Para este tema, a solução mais defensável é:

* TMDb como fonte principal;
* eventual OMDb como complemento;
* backend a centralizar os dados;
* frontend nunca falar diretamente com APIs externas.

Essa é uma inferência técnica, não uma exigência do PDF. 

---

## 9) IA no sistema

O PDF diz que a IA deve ser usada como apoio ao desenvolvimento, de forma crítica, mas não obriga a adicionar IA como funcionalidade do sistema. 

### Melhor decisão

IA no produto final é **opcional**. Para este sistema, só faria sentido se acrescentasse valor real, por exemplo:

* resumo automático de sinopse longa;
* explicação do motivo de uma recomendação;
* classificação de preferências do utilizador;
* assistente de descoberta de filmes.

Se o objetivo for maximizar estabilidade e tempo, a melhor decisão é **não depender de IA no core**.

## 11) Estratégia de recomendação

A lógica de recomendação pode começar simples e eficiente:

### Versão 1

* géneros preferidos;
* filmes mais bem avaliados pelo utilizador;
* filmes semelhantes aos favoritos;
* popularidade + afinidade de género.

### Versão 2

* peso por rating;
* peso por frequência de géneros;
* penalização por filmes já vistos;
* score final por recomendação.

### Modelo de score

Exemplo conceptual:
`score = afinidade_genro + media_ratings_similares + popularidade_normalizada - penalizacao_filmes_vistos`

Isto não é requisito do PDF; é a forma mais limpa de cumprir o objetivo do sistema com boa explicabilidade.

# 3.1 users

```sql id="50j17j"
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(120) NOT NULL,
    email VARCHAR(150) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role ENUM('user', 'admin') DEFAULT 'user',
    preferred_language VARCHAR(10) DEFAULT 'pt',
    theme ENUM('light', 'dark') DEFAULT 'dark',
```

```text id="y2p7vo"
score =
```
