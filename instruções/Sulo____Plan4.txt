## 1. Análise do tema à luz dos requisitos

O sistema deve ser um **Sistema de Recomendação de Filmes**, respeitando integralmente o caderno de encargos do Lab #04. As mesmas regras aplicam‑se: frontend Angular, backend PHP puro, SQL, autenticação completa, ≥3 tabelas com CRUD, API externa, interface responsiva, modo claro/escuro, i18n, exportação CSV/PDF e clara separação de responsabilidades. A natureza do problema (recomendação pessoal) **não justifica papéis de utilizador**, pelo que se mantém um único tipo de conta.

---

## 2. Escopo funcional

1. **Autenticação** – registo, login, logout, recuperação de senha via email.
2. **Pesquisa e exploração de filmes** – search box, listagem de filmes populares, detalhes (sinopse, elenco, trailer, avaliação média do TMDB). Tudo via API externa.
3. **Avaliação de filmes** (CRUD) – utilizador atribui nota de 1 a 5 estrelas. Pode criar, consultar, alterar ou remover a sua avaliação.
4. **Lista de favoritos / “Para ver”** (CRUD) – guardar filmes numa watchlist. Adicionar, listar e remover.
5. **Recomendações personalizadas** – com base nos géneros dos filmes bem avaliados (≥4 estrelas), sugerir filmes em alta nesses géneros que o utilizador ainda não avaliou.
6. **Exportação de dados** – exportar a lista de filmes avaliados ou recomendações em CSV.
7. **Preferências de UI** – alternância claro/escuro, seleção de idioma (PT/EN).

---

## 3. API externa escolhida

**The Movie Database (TMDB) API v3**  
- Gratuita, exige chave (registo simples).  
- Endpoints: `/movie/popular`, `/search/movie`, `/movie/{id}`, `/genre/movie/list`, `/discover/movie` (filtro por género).  
- Fornece todos os dados necessários (título, capa, sinopse, géneros, nota).  
- Limite generoso para um uso académico.

O backend funciona como **proxy** – o frontend nunca expõe a chave da API.

---

## 4. Arquitetura

```
[Angular SPA] -- HTTP/JSON --> [PHP Backend REST]
                                     |        |
                                  [MySQL]   [TMDB API]
                                     |
                              [Email SMTP]
```

- Autenticação via JWT.
- Camadas no backend: `Controllers` → `Services` → `Models/DB`.
- Recomendações calculadas no serviço `RecommendationService`, combinando dados do TMDB com as avaliações locais.

## 6. Base de dados

### 6.1 Tabelas (≥3, todas com CRUD)

```sql
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    language ENUM('pt','en') DEFAULT 'pt',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE ratings (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    movie_id INT NOT NULL,               -- id do TMDB
    rating TINYINT NOT NULL CHECK (rating BETWEEN 1 AND 5),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE KEY unique_user_movie (user_id, movie_id)
);

CREATE TABLE watchlist (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    movie_id INT NOT NULL,               -- id do TMDB
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE KEY unique_watchlist (user_id, movie_id)
);

CREATE TABLE password_resets (
    email VARCHAR(100) NOT NULL,
    token VARCHAR(255) NOT NULL,
    expires_at DATETIME NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_email (email)
);
```

- **users** – operações de registo e update (idioma).  
- **ratings** – CRUD completo (add, view, edit, delete).  
- **watchlist** – CRUD (add, view, delete).  
- `movie_id` é um `INT` vindo do TMDB; não guardamos metadados do filme localmente, mantendo a base leve.

## 9. Algoritmo de recomendação (simples e didático)

No backend, o método `getRecommendations(userId)`:
1. Obtém do TMDB a lista de todos os géneros (`/genre/movie/list`).
2. Consulta as avaliações do utilizador com nota ≥ 4 (baseado em `ratings`).
3. Para cada filme bem avaliado, vai ao TMDB buscar os `genre_ids` (pode ser cache).
4. Determina os 3 géneros mais frequentes.
5. Usa o endpoint `discover/movie` do TMDB com `with_genres`, ordenado por popularidade, e exclui filmes que o utilizador já avaliou ou que estão na watchlist (para evitar duplicados).
6. Retorna os 10 primeiros resultados.

Isto satisfaz o “sistema de recomendação” sem dependência de IA externa, e é inteiramente implementável com as tecnologias exigidas.

---

## 10. Exportação de dados

- `GET /api/export/ratings` → CSV com colunas: Título, Géneros (ou ids), Nota, Data.
- O controlador autentica o utilizador, carrega as avaliações do seu `user_id`, e para cada linha obtém do TMDB (ou de cache) o título actual. O frontend dispara o download via `window.open()` ou `Blob`.

---

## 11. Web scraping e IA no sistema

- **Web scraping:** Não será usado. O TMDB oferece uma API robusta e gratuita. O regulamento pede integração com API externa, e o scraping não é fiável nem respeita os termos de serviço.
- **Funcionalidade de IA:** O laboratório só exige que o processo de desenvolvimento seja assistido por IA. **Não é obrigatório incorporar IA no produto.** O sistema de recomendação baseado em géneros e avaliações é suficiente e adequado aos objetivos de Engenharia de Software. Não se incluirá, portanto, nenhum módulo de IA preditiva ou chatbot, a menos que o grupo o deseje como *extra* não vinculativo.
